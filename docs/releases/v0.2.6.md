# v0.2.6 — Document RAG Foundation

v0.2.6 adds the first end-to-end multi-source knowledge workflow: an admin uploads a document, the server parses and semantically chunks it, the shared retriever searches FAQ and document knowledge together, and chat answers from the highest-ranked sources.

Model endpoints, providers and names remain editable at runtime, but API keys are environment/deployment-secret only. Startup removes legacy plaintext key overrides from SQLite, and admin APIs never accept or return key material.

## Product Impact

- Admins can upload TXT, Markdown, text-layer PDF, and DOCX files from **Document Knowledge**.
- Each accepted document has an explicit `pending`, `ready`, or `failed` lifecycle, a stable failure code, enable/disable controls, retry, paginated chunk preview, and delete.
- Exact FAQ matches remain deterministic. Other queries use the global FAQ/document Top 5, with at most three untrusted excerpts supplied to the LLM.
- Without paid model keys, the local path returns the highest-ranked document name and original excerpt instead of generating a summary.
- Knowledge-review snapshots can preserve document chunk provenance while conversion still creates FAQs only.

## Compatibility And Data

- Existing `intent`, `faq`, `token`, `escalate`, `done`, and `error` SSE event types are unchanged. The `faq` event still contains FAQ entries only.
- `documents` and `document_chunks` are additive SQLite tables. Document deletion cascades to chunks; existing FAQ and conversation tables are not rewritten.
- Original files are stored under UUID names in the private upload directory. API responses omit storage paths, SHA-256 values, embeddings, and raw parser exceptions.
- FAQ index rebuilds refresh only the FAQ namespace and preserve document entries in the shared in-memory `VectorStore`.

## Limits And Security

- Upload validation checks extension, MIME, magic bytes, UTF-8, size, and duplicate SHA-256 before persistence. Limits are 10 MB per file, 200,000 extracted characters, 2,000 semantic units, and 300 final chunks.
- Document parsing and embedding are synchronous inside the Express process. Resource limits reduce exposure, but worker isolation is deferred.
- PDF support requires a text layer. Encrypted, damaged, and scanned PDFs fail explicitly; OCR and image understanding are not included.
- v0.2.6 does not add citations, page jumps, no-evidence refusal, reranking, web ingestion, document versions, tenant-separated knowledge bases, or an external vector database.
