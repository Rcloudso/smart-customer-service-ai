# v0.2.6 Release Evidence

## Scenario

1. Sign in as an administrator and upload `refund-policy.md` from **Document Knowledge**.
2. Confirm the document reaches `ready`, inspect the parsed chunk, and keep it enabled.
3. Ask the customer chat for the unique refund policy sentence with no model keys configured.
4. Confirm the answer names `refund-policy.md` and returns its original three-business-day excerpt.
5. Rate the answer negatively and confirm Knowledge Review preserves the document/chunk retrieval snapshot.
6. Delete the document and confirm its file, database row, chunks, and in-memory index entry are removed.

## Automated Evidence

- `EMBED_PROVIDER=other npm test`: passed route failure handling, knowledge review, four-format parser checks, semantic chunking, document lifecycle/index consistency, generic retrieval, prompt isolation, snapshot compatibility, document HTTP/chat integration, regression checks, and server/client TypeScript.
- `EMBED_PROVIDER=other npm run eval:faq`: 11 cases; Top1 100%, Top3 100%, no-match 100%; source distribution `hybrid: 9`, `vector: 2`.
- `EMBED_PROVIDER=other npm run eval:document`: 12 cases across TXT, Markdown, PDF, and DOCX; semantic-v1 Top3 100%, MRR 1.000; structure baseline Top3 100%, MRR 1.000.
- `PLAYWRIGHT_CHANNEL=chromium npm run test:e2e`: 15/15 API and Chromium tests passed, including document auth/validation/privacy/lifecycle, FAQ SSE compatibility, no-key source fallback, bilingual/theme/mobile UI, and the upload-to-answer workflow.
- `CAPTURE_RELEASE_EVIDENCE=1 PLAYWRIGHT_CHANNEL=chromium npx playwright test --project=web-chrome --grep "admin uploads a document"`: passed and regenerated the final screenshots below.
- `EMBED_PROVIDER=other npm run build`: server TypeScript and Vite production build passed. The existing non-blocking PostCSS module-type warning remains.
- `git diff --check`: passed.

## Security Review Evidence

- Upload responses and logs do not expose storage paths, document hashes, embeddings, or raw parser errors.
- Prompt tests verify that document-provided `<knowledge>` delimiters are escaped and only the top three results enter the prompt as explicitly untrusted material.
- `npm audit --omit=dev` reached the official npm advisory service and reported existing findings through `bcrypt`/`tar`, `uuid`, and `xlsx`; neither `mammoth` nor `pdf-parse` appeared in the findings. The `xlsx` findings currently have no upstream fix and remain a repository-wide known risk outside this document-RAG slice.
- Docker/Node 20 verification was not runnable on this workstation because the `docker` command and a local Node 20 runtime are unavailable. CI remains configured for Node 20 with `npm ci`, both retrieval evaluations, E2E, and build.

## Known Limits

- Parsing and embedding run synchronously in the Express process behind administrator authorization and fixed resource caps; worker isolation is deferred.
- PDF requires a text layer. OCR, image knowledge, citations/page jumps, reranking, no-evidence refusal, web ingestion, and document versioning remain out of scope.
- SQLite plus the in-memory vector index remains the intended small-deployment path; no external vector database is required for v0.2.6.
