# v0.2.7 — Grounding, Citations & Refusal

v0.2.7 makes the answer boundary explicit. The server now decides whether to
return a high-confidence FAQ, generate after a retrieved source clears the
initial threshold, or refuse before it calls the answer-generation stream.
That decision and the supporting source snapshot are
persisted with the assistant message and restored with conversation history.

## Product Impact

- Customer answers show an answer mode and grounding status in Chinese and
  English.
- High-confidence keyword/hybrid FAQ answers remain deterministic and keep
  their FAQ reference after a page reload.
- Generated answers persist up to three retrieved FAQ or document sources with document,
  chunk, and optional page metadata.
- Missing or weak evidence produces a deterministic refusal without calling the
  answer-generation stream. Configured intent classification may still use a model.
- Duplicate direct FAQs with materially different normalized answers and
  recognized private-state/action requests are refused and escalated for human review.
- Explicit human requests are handled before generation and do not create a
  partial model response.

## Compatibility And Data

- Existing `intent`, `faq`, `token`, `escalate`, `done`, and `error` SSE event
  names are unchanged.
- `answerMode`, `groundingStatus`, `groundingReason`, and `knowledgeSources` are
  optional additions to `done.content`.
- Assistant messages receive additive, idempotently migrated columns for the
  answer mode and grounding decision. Existing rows remain readable.
- Conversation detail now returns the already-persisted retrieval snapshot,
  fixing source cards that previously disappeared after history restoration.

## Reliability And Architecture

- Model and embedding calls use abortable, bounded timeouts.
- A streaming model call is never retried after a token has been emitted, which
  prevents replayed partial answers.
- JSON and SSE mutations accept an optional `Idempotency-Key`. Matching retries
  replay the SQLite-backed response, while different payloads and concurrent
  in-flight duplicates return `409`.
- Customer and admin mutation controls use synchronous single-flight guards,
  closing the gap before React loading state disables the button.
- Multipart uploads are intentionally outside generic replay: document content
  keeps SHA-256 duplicate protection and import/upload controls reject rapid
  re-entry.
- The API exposes separate liveness and readiness probes and performs bounded
  graceful shutdown for HTTP connections and SQLite.
- [ARCHITECTURE.md](../../ARCHITECTURE.md) documents the current topology,
  trust boundaries, failure behavior, and evidence-based scaling triggers.

## Limits

- The grounding thresholds are initial deterministic safeguards and will be
  calibrated with versioned mixed-source evaluations in v0.2.8.
- Conflict detection is limited to duplicate normalized direct-FAQ questions
  with different answers.
- Order, logistics, refund, cancellation, and address changes are not connected
  to business tools in this release; recognized private-state/action requests
  are refused and escalated.
- This release does not add Redis, background workers, an external vector
  database, multi-replica coordination, OCR, or web ingestion.
- Idempotency records are deployment-local and retained for 24 hours; they do
  not coordinate external business APIs or multiple independent API stores.
- Source cards are retrieval snapshots, not claim-level citation alignment or
  post-generation factual entailment checks.
